LostLink — Forgot Password Implementation
Date: 2026-04-20
=======================================================

OVERVIEW
--------
Two-step password reset flow:
  1. User requests a reset link via their email address.
  2. Supabase emails them a one-time link.
  3. They click it, land on /reset-password, set a new password.

=======================================================
SUPABASE CONFIGURATION REQUIRED (do this first)
=======================================================

1. Go to your Supabase project dashboard.
2. Navigate to: Authentication → URL Configuration
3. Under "Redirect URLs", add ALL of the following:

   http://localhost:5173/reset-password       ← local dev
   https://csdp490.vercel.app/reset-password  ← Vercel production
   https://csdp490.qr-manager.net/reset-password  ← custom domain (if used)

   Without these, Supabase will refuse to send the reset email and
   silently fail or redirect to a blocked URL error page.

4. No new database tables are needed for this feature.

=======================================================
ENDPOINT DETAILS
=======================================================

POST /auth/forgot-password
  Rate limit: 3/minute (slowapi, IP-based)
  Body: { "email": "user@example.com" }

  Calls auth_supabase.auth.reset_password_for_email() with
  redirect_to = FRONTEND_URL + "/reset-password".

  Always returns 200 with the same message regardless of whether the
  email exists — this prevents attackers from enumerating registered
  emails by observing different error responses.

  Response: { "message": "If that email is registered, you'll receive
              a reset link shortly." }

POST /auth/reset-password
  No rate limit (token is single-use by design)
  Body: { "token": "<recovery token>", "new_password": "..." }

  1. Validates new_password length >= 8, else 400.
  2. Calls auth_supabase.auth.get_user(token) to verify the token is
     valid and get the user's ID. Raises 401 if invalid/expired.
  3. Calls db_supabase.auth.admin.update_user_by_id(uid, {"password": pw})
     using the service role key to update the password.
  4. Returns 200 { "message": "Password updated successfully" }.

  The recovery token is a short-lived JWT issued by Supabase when the
  reset email is sent. It expires after 1 hour and is single-use.

=======================================================
FRONTEND PAGES
=======================================================

/forgot-password  (src/pages/ForgotPassword.jsx)
  - Email input form, submit sends POST /auth/forgot-password
  - On success: replaces form with a "check your inbox" message
  - Same success message shown regardless of whether email is registered
    (mirrors the backend anti-enumeration behaviour)
  - "Back to login" link always visible
  - Rate limiting on the backend means the button can only be hit 3x/min

/reset-password  (src/pages/ResetPassword.jsx)
  - On mount, reads window.location.hash and extracts:
      access_token — the recovery JWT
      type         — must equal "recovery"
    If either is missing/wrong: shows an "invalid link" error with a
    link to request a fresh one.
  - After reading the token, clears it from the URL bar with
    window.history.replaceState() so it's not visible or re-usable
    from browser history.
  - Form: new password + confirm password fields
  - Client-side validation: length >= 8, passwords match
  - Submits POST /auth/reset-password {token, new_password}
  - On success: shows confirmation banner, redirects to /login after 3s
  - On failure: shows the error from the backend (invalid/expired link)

=======================================================
HOW THE RESET LINK WORKS
=======================================================

When Supabase sends the reset email, the link looks like:
  https://yourapp.com/reset-password#access_token=XXX&expires_in=3600
  &refresh_token=YYY&token_type=bearer&type=recovery

The hash fragment (#...) is never sent to the server — it stays in the
browser. The React app at /reset-password reads it via window.location.hash
and parses it with URLSearchParams. The token (access_token) is then sent
to the backend via POST /auth/reset-password.

This means no server-side session handling is needed for the reset flow.
The token itself IS the credential — the backend validates it with Supabase
Auth before accepting the new password.

=======================================================
SECURITY NOTES
=======================================================

- Email enumeration is prevented: /auth/forgot-password always returns
  200 with the same message.
- Reset tokens expire after 1 hour (Supabase-controlled).
- Tokens are single-use: Supabase invalidates the token after
  get_user() validates it for a password update.
- The token is cleared from the URL bar on mount so it doesn't appear
  in browser history or leak via the Referer header.
- Minimum password length of 8 characters enforced on both client
  and backend.
- The 3/minute rate limit on /auth/forgot-password prevents abuse of
  Supabase's email sending quota.
